0.721 s / 0.757 s / 0.719 s wall time — all three reported, not just the fastest, the same
disclosure this report already applies to §3's single sample. `--hook` prints nothing on success
(no `typer.echo` on that branch), matching its documented "silent" behavior — confirmed by the
empty stdout above. A follow-up `--dry-run` after all three (not reprinted — byte-identical to
the block two above) confirms nothing moved: still `vectors: nothing`, `paths: nothing`, budget
still 0 calls / $0.0000 used.

**A real finding worth stating plainly, not smoothing over: the embedder is not actually skipped
here.** `_run_peers` (`wikiforge/ops/maintain.py`) calls `ctx.embedder().model` for every
registered peer to compute its compatibility verdict, and because the peer registry is
machine-global with a permanent entry, `peers` has work — probe `True` — on every wiki on this
machine, `rss` included, every single run, independent of `vectors`/`paths` state. So
`JobContext.embedder()` *is* called on all three timed runs above; the probe-first design does
not make this wiki's `maintain --hook` embedder-free. It does not, however, reintroduce the ~9 s
cost: `.model` is a plain stored attribute (`CachedEmbeddingProvider.model` in
`wikiforge/embed/provider.py` and `LocalEmbeddingProvider.model` in `wikiforge/embed/local.py`
both just return an already-set string), and the actual fastembed/sentence-transformers load
lives behind `_ensure_encoder()`, reached only from `.embed()` — which none of `vectors`,
`paths`, or `peers` call while there is nothing to embed. Building the wrapper is cheap; loading
the model is not; this run does only the former.

**Interpretation.** 0.72–0.76 s supports the "sub-second when nothing to backfill" expectation
Task 15's design implied, comfortably: it lands in the same range as this report's own pure-SQL
§3 baseline (0.904 s, `wiki why`, one peer, zero embedder involvement at all), and roughly 12×
faster than the ~9 s "before" figure. One honest caveat on comparability: `rss` runs
`BAAI/bge-small-en-v1.5` here, not the `intfloat/multilingual-e5-small`/torch-fallback model that
produced the 9 s figure on `~/wiki` — no wiki with paid work pending could be drained without
spending real budget, so this is a same-mechanism, different-model comparison, not an
identical-model one. The mechanism it verifies — `_ensure_encoder()` reached only from `.embed()`,
never called by any of this run's five jobs — is model-independent by construction, so the
conclusion (the regression is gone for the common case) holds regardless of which local model a
given wiki runs, even though the *number* above is specifically `rss`'s.
